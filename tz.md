# Требования к сайту продажи табачных изделий

## 1. Обзор проекта

Веб-платформа для продажи табачных изделий с соблюдением законодательства Республики Казахстан, которое требует ограничения открытого показа табачной продукции. Каталог товаров доступен только авторизованным пользователям после регистрации.

## 2. Технологический стек

- **База данных**: SQLite (локальное хранение на сервере)
- **Backend**: FastAPI (Python)
- **Frontend**: React с TypeScript
- **Аутентификация**: JWT токены
- **Стилизация**: Tailwind CSS

## 3. Функциональные требования

### 3.1 Система аутентификации и авторизации

**3.1.1 Регистрация пользователей**
- Пользователь может зарегистрироваться с указанием:
  - Email (уникальный)
  - Пароль (минимум 8 символов)
  - Имя и фамилия
  - Номер телефона
  - Дата рождения (должен быть 18+)
- Система проверяет возраст пользователя (минимум 18 лет)
- После регистрации отправляется email с подтверждением
- Пароли хранятся в зашифрованном виде (bcrypt)

**Критерии приемки:**
- Регистрация невозможна для лиц младше 18 лет
- Email должен быть уникальным в системе
- Пароль должен соответствовать требованиям безопасности
- Успешная регистрация создает запись в БД и отправляет email

**3.1.2 Вход в систему**
- Пользователь входит используя email и пароль
- При успешном входе генерируется JWT токен
- Токен имеет срок действия 24 часа
- Возможность "Запомнить меня" (refresh token на 30 дней)

**Критерии приемки:**
- Неавторизованные пользователи не могут получить доступ к каталогу
- JWT токен корректно генерируется и валидируется
- Неверные учетные данные возвращают понятное сообщение об ошибке

**3.1.3 Восстановление пароля**
- Пользователь может запросить сброс пароля через email
- Генерируется временная ссылка для сброса (действительна 1 час)
- Пользователь устанавливает новый пароль

**Критерии приемки:**
- Ссылка для сброса пароля истекает через 1 час
- Старый пароль перестает работать после установки нового

### 3.2 Управление каталогом товаров (Админ-панель)

**3.2.1 Добавление товара**
- Администратор может добавить новый товар с полями:
  - Название товара
  - Описание (краткое и полное)
  - Категория (сигареты, сигары, табак для кальяна, вейп-продукция, аксессуары)
  - Бренд/производитель
  - Цена (в тенге)
  - Количество на складе
  - Изображения (минимум 1, максимум 5)
  - Характеристики (крепость, вкус, объем и т.д.)
  - SKU (артикул)
  - Статус (активен/неактивен)

**Критерии приемки:**
- Все обязательные поля должны быть заполнены
- Изображения загружаются и сохраняются на сервере
- Товар появляется в каталоге только после сохранения

**3.2.2 Редактирование товара**
- Администратор может изменить любые данные существующего товара
- История изменений сохраняется (кто и когда изменил)
- Возможность деактивировать товар без удаления

**Критерии приемки:**
- Изменения сохраняются корректно
- Деактивированные товары не отображаются в каталоге для покупателей
- История изменений доступна администратору

**3.2.3 Удаление товара**
- Администратор может удалить товар
- Мягкое удаление (товар помечается как удаленный, но остается в БД)
- Удаленные товары не отображаются в каталоге

**Критерии приемки:**
- Удаленный товар не виден покупателям
- Данные о товаре сохраняются в БД для истории заказов

**3.2.4 Управление категориями**
- Администратор может создавать, редактировать и удалять категории
- Каждая категория имеет название и описание
- Возможность создания подкатегорий

**Критерии приемки:**
- Категории корректно отображаются в каталоге
- Удаление категории не удаляет товары (они переходят в "Без категории")

### 3.3 Каталог товаров (для авторизованных пользователей)

**3.3.1 Просмотр каталога**
- Доступ к каталогу только после авторизации
- Неавторизованные пользователи видят страницу с требованием войти/зарегистрироваться
- Отображение товаров в виде сетки с карточками
- Каждая карточка содержит:
  - Изображение товара
  - Название
  - Цену
  - Краткое описание
  - Кнопку "Подробнее"
  - Кнопку "В корзину"

**Критерии приемки:**
- Неавторизованные пользователи не могут видеть товары
- Карточки товаров отображаются корректно
- Клик по карточке открывает детальную страницу товара

**3.3.2 Фильтрация и сортировка**
- Фильтрация по:
  - Категориям
  - Бренду
  - Ценовому диапазону
  - Наличию на складе
  - Характеристикам (крепость, вкус и т.д.)
- Сортировка по:
  - Цене (возрастание/убывание)
  - Названию (А-Я, Я-А)
  - Новизне
  - Популярности

**Критерии приемки:**
- Фильтры работают корректно и можно комбинировать несколько фильтров
- Сортировка применяется мгновенно
- Количество найденных товаров отображается

**3.3.3 Поиск товаров**
- Полнотекстовый поиск по названию, описанию, бренду
- Автодополнение при вводе
- Поиск работает с учетом морфологии (русский и казахский языки)
- Отображение результатов поиска в реальном времени

**Критерии приемки:**
- Поиск находит релевантные товары
- Автодополнение работает быстро (< 300ms)
- Пустой результат поиска показывает понятное сообщение

**3.3.4 Детальная страница товара**
- Полная информация о товаре:
  - Галерея изображений (с возможностью увеличения)
  - Название и бренд
  - Полное описание
  - Цена
  - Наличие на складе
  - Все характеристики
  - SKU
- Кнопка "Добавить в корзину"
- Выбор количества товара
- Блок "Похожие товары"
- Отзывы покупателей (если есть)

**Критерии приемки:**
- Вся информация отображается корректно
- Галерея изображений работает плавно
- Нельзя добавить в корзину больше товара, чем есть на складе
- Похожие товары релевантны текущему товару

### 3.4 Корзина покупок

**3.4.1 Добавление товаров в корзину**
- Пользователь может добавить товар в корзину с любой страницы каталога
- Указание количества товара
- Визуальное подтверждение добавления (уведомление)
- Счетчик товаров в корзине в шапке сайта

**Критерии приемки:**
- Товар добавляется в корзину мгновенно
- Счетчик обновляется автоматически
- Нельзя добавить больше товара, чем есть на складе

**3.4.2 Просмотр корзины**
- Список всех товаров в корзине с:
  - Изображением
  - Названием
  - Ценой за единицу
  - Количеством (с возможностью изменения)
  - Общей стоимостью позиции
  - Кнопкой удаления
- Общая сумма заказа
- Кнопка "Оформить заказ"
- Кнопка "Продолжить покупки"

**Критерии приемки:**
- Корзина корректно отображает все добавленные товары
- Изменение количества пересчитывает сумму
- Удаление товара работает мгновенно

**3.4.3 Сохранение корзины**
- Корзина сохраняется для авторизованного пользователя
- При повторном входе корзина восстанавливается
- Корзина очищается после успешного оформления заказа

**Критерии приемки:**
- Корзина сохраняется между сеансами
- Данные корзины не теряются при выходе из системы

### 3.5 Оформление заказа

**3.5.1 Процесс оформления**
- Шаг 1: Проверка корзины
- Шаг 2: Указание адреса доставки
  - Город
  - Улица, дом, квартира
  - Комментарий к адресу
- Шаг 3: Выбор способа доставки
  - Курьерская доставка
  - Самовывоз из магазина
- Шаг 4: Выбор способа оплаты
  - Наличными при получении
  - Картой онлайн
  - Kaspi перевод
- Шаг 5: Подтверждение заказа

**Критерии приемки:**
- Все шаги проходят последовательно
- Нельзя перейти к следующему шагу без заполнения текущего
- Данные сохраняются при переходе между шагами

**3.5.2 Создание заказа**
- После подтверждения создается заказ в БД со статусом "Новый"
- Генерируется уникальный номер заказа
- Отправляется email с подтверждением заказа
- Уменьшается количество товара на складе
- Корзина очищается

**Критерии приемки:**
- Заказ создается с корректными данными
- Email отправляется успешно
- Остатки на складе обновляются
- Пользователь видит страницу с подтверждением и номером заказа

### 3.6 Личный кабинет пользователя

**3.6.1 Профиль пользователя**
- Просмотр и редактирование личных данных:
  - Имя, фамилия
  - Email
  - Телефон
  - Дата рождения
- Изменение пароля
- Управление адресами доставки (добавление, редактирование, удаление)

**Критерии приемки:**
- Изменения сохраняются корректно
- При изменении email требуется подтверждение
- Старый пароль требуется для установки нового

**3.6.2 История заказов**
- Список всех заказов пользователя
- Для каждого заказа отображается:
  - Номер заказа
  - Дата оформления
  - Статус
  - Общая сумма
  - Список товаров
- Возможность просмотра детальной информации о заказе
- Возможность повторить заказ (добавить те же товары в корзину)

**Критерии приемки:**
- История заказов отображается корректно
- Детальная информация доступна по клику
- Повторный заказ добавляет товары в корзину

**3.6.3 Избранное**
- Пользователь может добавлять товары в избранное
- Просмотр списка избранных товаров
- Удаление из избранного
- Быстрое добавление в корзину из избранного

**Критерии приемки:**
- Избранное сохраняется между сеансами
- Добавление/удаление работает мгновенно

### 3.7 Административная панель

**3.7.1 Управление заказами**
- Просмотр всех заказов с фильтрацией по статусу
- Изменение статуса заказа:
  - Новый
  - Подтвержден
  - В обработке
  - Передан в доставку
  - Доставлен
  - Отменен
- Просмотр детальной информации о заказе
- Печать накладной
- Отправка уведомлений клиенту об изменении статуса

**Критерии приемки:**
- Все заказы отображаются корректно
- Статус изменяется и сохраняется
- Email уведомления отправляются при изменении статуса

**3.7.2 Управление пользователями**
- Просмотр списка всех пользователей
- Поиск пользователей по email, имени, телефону
- Просмотр профиля пользователя
- Просмотр истории заказов пользователя
- Блокировка/разблокировка пользователя

**Критерии приемки:**
- Список пользователей отображается с пагинацией
- Поиск работает корректно
- Заблокированный пользователь не может войти в систему

**3.7.3 Аналитика и отчеты**
- Дашборд с ключевыми метриками:
  - Количество заказов за период
  - Общая сумма продаж
  - Средний чек
  - Количество новых пользователей
  - Топ-10 популярных товаров
- Графики продаж по дням/неделям/месяцам
- Экспорт отчетов в CSV/Excel

**Критерии приемки:**
- Метрики рассчитываются корректно
- Графики отображаются правильно
- Экспорт работает без ошибок

### 3.8 Система уведомлений

**3.8.1 Email уведомления**
- Подтверждение регистрации
- Восстановление пароля
- Подтверждение заказа
- Изменение статуса заказа
- Уведомление о поступлении товара (если был недоступен)

**Критерии приемки:**
- Все email отправляются корректно
- Шаблоны писем оформлены профессионально
- Ссылки в письмах работают

## 4. Нефункциональные требования

### 4.1 Безопасность

**4.1.1 Защита данных**
- Все пароли хранятся в зашифрованном виде (bcrypt)
- HTTPS для всех соединений
- JWT токены для аутентификации
- Защита от SQL-инъекций (использование ORM)
- Защита от XSS атак
- CSRF токены для форм

**Критерии приемки:**
- Пароли невозможно восстановить из БД
- Все запросы идут через HTTPS
- Токены корректно валидируются

**4.1.2 Соблюдение законодательства РК**
- Каталог табачных изделий доступен только авторизованным пользователям 18+
- Проверка возраста при регистрации
- Предупреждение о вреде курения на всех страницах с товарами
- Запрет на рекламу табачных изделий (нет SEO оптимизации для каталога)

**Критерии приемки:**
- Неавторизованные пользователи не видят товары
- Регистрация невозможна для лиц младше 18 лет
- Предупреждения отображаются на всех страницах каталога

### 4.2 Производительность

**4.2.1 Скорость загрузки**
- Главная страница загружается за < 2 секунды
- Страница каталога загружается за < 3 секунды
- Детальная страница товара загружается за < 2 секунды
- API запросы обрабатываются за < 500ms

**Критерии приемки:**
- Время загрузки соответствует требованиям
- Изображения оптимизированы и загружаются быстро

**4.2.2 Оптимизация**
- Ленивая загрузка изображений
- Кэширование статических ресурсов
- Минификация CSS и JavaScript
- Сжатие изображений
- Пагинация для больших списков (20 товаров на страницу)

**Критерии приемки:**
- Изображения загружаются по мере прокрутки
- Статические файлы кэшируются браузером
- Размер страниц оптимизирован

### 4.3 Удобство использования (UX/UI)

**4.3.1 Адаптивный дизайн**
- Корректное отображение на всех устройствах:
  - Desktop (1920px и выше)
  - Laptop (1366px - 1920px)
  - Tablet (768px - 1366px)
  - Mobile (320px - 768px)
- Touch-friendly интерфейс для мобильных устройств

**Критерии приемки:**
- Сайт корректно отображается на всех разрешениях
- Все элементы доступны и кликабельны на мобильных

**4.3.2 Интуитивная навигация**
- Понятная структура меню
- Хлебные крошки на всех страницах
- Поиск доступен с любой страницы
- Корзина и профиль доступны из шапки сайта
- Кнопка "Наверх" при прокрутке

**Критерии приемки:**
- Пользователь может легко найти нужный раздел
- Навигация работает интуитивно

**4.3.3 Многоязычность**
- Поддержка русского и казахского языков
- Переключение языка доступно на всех страницах
- Выбранный язык сохраняется для пользователя

**Критерии приемки:**
- Весь интерфейс переведен на оба языка
- Переключение работает мгновенно
- Язык сохраняется между сеансами

### 4.4 Масштабируемость

**4.4.1 База данных**
- SQLite для начального этапа (до 10,000 товаров)
- Возможность миграции на PostgreSQL при росте
- Индексы на часто используемых полях
- Оптимизированные запросы

**Критерии приемки:**
- БД работает стабильно с текущим объемом данных
- Запросы выполняются быстро

**4.4.2 Архитектура**
- Модульная структура кода
- Разделение на слои (API, бизнес-логика, данные)
- Возможность горизонтального масштабирования

**Критерии приемки:**
- Код организован логически
- Легко добавлять новые функции

### 4.5 Надежность

**4.5.1 Обработка ошибок**
- Все ошибки логируются
- Пользователю показываются понятные сообщения об ошибках
- Graceful degradation при недоступности сервисов
- Автоматическое восстановление после сбоев

**Критерии приемки:**
- Ошибки не приводят к падению приложения
- Логи содержат достаточно информации для отладки

**4.5.2 Резервное копирование**
- Ежедневное автоматическое резервное копирование БД
- Хранение резервных копий за последние 30 дней
- Возможность восстановления из резервной копии

**Критерии приемки:**
- Резервные копии создаются автоматически
- Восстановление работает корректно

## 5. Структура базы данных

### 5.1 Таблицы

**users** - Пользователи
- id (INTEGER, PRIMARY KEY)
- email (TEXT, UNIQUE, NOT NULL)
- password_hash (TEXT, NOT NULL)
- first_name (TEXT, NOT NULL)
- last_name (TEXT, NOT NULL)
- phone (TEXT)
- birth_date (DATE, NOT NULL)
- is_active (BOOLEAN, DEFAULT TRUE)
- is_admin (BOOLEAN, DEFAULT FALSE)
- created_at (TIMESTAMP)
- updated_at (TIMESTAMP)

**categories** - Категории товаров
- id (INTEGER, PRIMARY KEY)
- name_ru (TEXT, NOT NULL)
- name_kk (TEXT, NOT NULL)
- description_ru (TEXT)
- description_kk (TEXT)
- parent_id (INTEGER, FOREIGN KEY -> categories.id)
- created_at (TIMESTAMP)

**products** - Товары
- id (INTEGER, PRIMARY KEY)
- sku (TEXT, UNIQUE, NOT NULL)
- name_ru (TEXT, NOT NULL)
- name_kk (TEXT, NOT NULL)
- description_short_ru (TEXT)
- description_short_kk (TEXT)
- description_full_ru (TEXT)
- description_full_kk (TEXT)
- category_id (INTEGER, FOREIGN KEY -> categories.id)
- brand (TEXT)
- price (DECIMAL, NOT NULL)
- stock_quantity (INTEGER, DEFAULT 0)
- is_active (BOOLEAN, DEFAULT TRUE)
- is_deleted (BOOLEAN, DEFAULT FALSE)
- created_at (TIMESTAMP)
- updated_at (TIMESTAMP)

**product_images** - Изображения товаров

## 14. Интеграция с 1С (будущая функциональность)

### 14.1 Обзор интеграции

Планируется двусторонняя интеграция с системой 1С:Предприятие для синхронизации данных о товарах, остатках, заказах и клиентах.

### 14.2 Направления интеграции

**14.2.1 Из 1С в веб-сайт (импорт):**
- Товары и их характеристики
- Остатки на складе
- Цены
- Категории товаров
- Изображения товаров
- Статусы заказов

**14.2.2 Из веб-сайта в 1С (экспорт):**
- Новые заказы
- Данные клиентов
- Информация о платежах
- История изменений заказов

### 14.3 Функциональные требования интеграции

**14.3.1 Синхронизация товаров**
- Автоматическая синхронизация каталога товаров из 1С
- Обновление цен в режиме реального времени или по расписанию
- Синхронизация остатков товаров
- Обновление характеристик и описаний
- Загрузка изображений товаров из 1С

**Критерии приемки:**
- Товары из 1С корректно импортируются в БД сайта
- Изменения в 1С отражаются на сайте в течение заданного интервала
- Конфликты данных обрабатываются корректно (приоритет у 1С)

**14.3.2 Синхронизация остатков**
- Автоматическое обновление количества товара на складе
- Резервирование товара при оформлении заказа
- Снятие резерва при отмене заказа
- Обновление остатков после отгрузки

**Критерии приемки:**
- Остатки на сайте соответствуют данным в 1С
- Невозможно заказать товар, которого нет в наличии
- Резервы корректно учитываются

**14.3.3 Выгрузка заказов в 1С**
- Автоматическая передача новых заказов в 1С
- Передача полной информации о заказе:
  - Данные клиента
  - Состав заказа
  - Адрес доставки
  - Способ оплаты и доставки
  - Комментарии
- Создание документа "Заказ покупателя" в 1С

**Критерии приемки:**
- Заказы передаются в 1С без потерь данных
- В 1С создается корректный документ заказа
- Ошибки передачи логируются и обрабатываются

**14.3.4 Синхронизация статусов заказов**
- Обновление статуса заказа на сайте при изменении в 1С
- Уведомление клиента об изменении статуса
- Синхронизация номера накладной и трек-номера

**Критерии приемки:**
- Статусы синхронизируются в обе стороны
- Клиент получает актуальную информацию о заказе

**14.3.5 Синхронизация клиентов**
- Создание карточки клиента в 1С при первом заказе
- Обновление контактных данных клиента
- Синхронизация истории покупок
- Учет бонусов и скидок (если применимо)

**Критерии приемки:**
- Клиенты из сайта корректно создаются в 1С
- Данные клиентов актуальны в обеих системах

### 14.4 Технические требования интеграции

**14.4.1 Протокол обмена данными**
- REST API для обмена данными
- Альтернатива: ComConnector для прямого подключения к 1С
- Формат данных: JSON или XML
- Аутентификация: API ключи или OAuth 2.0

**14.4.2 Архитектура интеграции**
- Создание отдельного модуля интеграции в backend
- Очередь задач для асинхронной обработки (Celery + Redis)
- Логирование всех операций обмена
- Механизм повторных попыток при ошибках

**14.4.3 Расписание синхронизации**
- Товары и цены: каждые 15 минут
- Остатки: каждые 5 минут
- Заказы: немедленно после создания
- Статусы заказов: каждые 10 минут
- Клиенты: при создании/изменении

**14.4.4 Обработка ошибок**
- Логирование всех ошибок интеграции
- Уведомление администратора о критических ошибках
- Автоматические повторные попытки (до 3 раз)
- Ручная синхронизация через админ-панель
